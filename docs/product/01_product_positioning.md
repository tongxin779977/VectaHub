# VectaHub 产品定位

> 版本: 6.0.0 | 最后更新: 2026-05-02

---

## 一句话定位

**VectaHub 是一个用自然语言定义本地自动化工作流的工具，安全执行，可复用。**

---

## 核心问题

开发者每天面对大量重复的本地自动化任务：

- "把 build 产物压缩后上传到 CDN"
- "批量重命名这些图片，然后压缩"
- "跑完测试，如果通过了就部署"
- "每天凌晨备份数据库到外接硬盘"

现有的解决方案要么需要写脚本（Makefile/Bash），要么需要手动点 GUI。VectaHub 让用户**说人话，自动编排工作流，安全执行**。

---

## 与 OpenCLI 的关系

VectaHub 不跟 OpenCLI 竞争，而是**互补**。

| 维度 | OpenCLI | VectaHub |
|------|---------|----------|
| **核心能力** | 把网站/浏览器变成 CLI | 用自然语言编排本地工作流 |
| **执行环境** | Chrome 登录会话 | 本地沙盒 |
| **AI 集成** | AI Agent 操作浏览器 | LLM 理解意图 → 生成工作流 |
| **确定性** | 同一命令，同一输出 | 灵活编排，条件/循环/并行 |
| **适用场景** | 刷网站、爬数据、网页操作 | 文件处理、脚本编排、CI/CD |
| **运行时 LLM 成本** | 零 | 按需调用 |

**互补关系**：
```
OpenCLI 负责"手"（网站/浏览器/CLI 调用）
VectaHub 负责"脑"（理解意图 → 编排工作流 → 安全执行）
```

VectaHub 的工作流可以直接调用 OpenCLI 作为步骤执行：

```yaml
steps:
  - id: step1
    type: opencli
    site: hackernews
    command: top
    args: ["--limit", "10"]

  - id: step2
    type: exec
    cli: node
    args: ["-e", "process data from step1..."]
```

## 目标用户

1. **开发者**：不想写 Makefile/Bash 脚本，希望用自然语言定义任务
2. **运维人员**：需要安全执行自动化任务（沙箱隔离、权限控制）
3. **AI Agent 集成者**：需要一个安全的 CLI 执行引擎

---

## 核心价值主张

| 价值 | 说明 | 现状 |
|------|------|------|
| **自然语言** | 说人话，自动生成工作流 | 需要 LLM（已实现客户端，待接入核心路径） |
| **工作流编排** | 条件/循环/并行/依赖 | 已实现 |
| **安全执行** | 三层安全检测 + 多平台沙箱 | 已实现 |
| **可复用** | 保存为 YAML，随时复用 | 已实现 |
| **OpenCLI 集成** | 调用 90+ 网站适配器 | 已实现基础，待扩展 |

## 三步实施路径

### 第一步：LLM 接入核心路径（当前优先级）

- 将已有的 LLM 客户端接入 `run` 命令
- 实现 LLM → 意图解析 → 工作流生成
- 无 LLM 时降级为关键词匹配

### 第二步：OpenCLI 编排层

- 扩展工作流模板，覆盖更多 opencli 场景
- 优化 opencli 步骤的错误处理和输出传递
- 实现 opencli 工具发现机制

### 第三步：企业安全场景

- 强化审计日志和权限管理
- 实现定时任务（cron）
- Web UI 管理面板

---

## 诚实的当前状态

| 能力 | 真实状态 |
|------|----------|
| 自然语言理解 | 目前是关键词匹配，LLM 客户端已实现但未接入核心路径 |
| 工作流编排 | 核心功能完整（exec/if/for_each/parallel/opencli），但条件表达式有限 |
| 安全沙箱 | 三层检测有效，但降级模式（无 sandbox-exec/bwrap）等于普通执行 |
| CLI 工具 | 4 个工具（git/npm/docker/curl），只是元数据注册，非重新实现 |
| LLM 集成 | API 客户端完整实现，支持 OpenAI/Anthropic/Ollama，但只在 generate 命令中使用 |

**不是空壳，但是半成品。** 核心引擎有真实价值，但"自然语言"能力需要 LLM 接入才能从假变真。
